# AI-Generated-Code Security Review Checklist

A manual audit checklist for reviewing AI-written (or AI-assisted) code in this
repo. It targets the failure modes that LLM coding tools are _documented_ to
produce, grounded in the threads, issues, and write-ups linked under
[Sources](#sources).

## Why this exists

Parts of this codebase are AI-generated by design (`@anthropic-ai/sdk` +
`.github/workflows/content-pipeline.yml`), and the app handles **real PII** —
names, phones, emails, street addresses — flowing to HighLevel, Twilio, and
Resend, behind an OAuth-gated admin panel with two inbound webhooks. That is
exactly the surface where AI-written code is shown to fail: research from
Stanford/UIUC found developers using AI assistants produced **less** secure code
while feeling **more** confident it was secure, and a scan of 100 AI-generated
repos found **67/100 had at least one critical vulnerability** (see Sources).
LLMs generate one file at a time and lose the cross-file picture (auth flow ↔
route ↔ middleware), so a green diff is not a safe diff.

## How to use this

Run this checklist on **every AI-authored or AI-assisted PR** that touches:

- `src/pages/api/**` (serverless endpoints, webhooks, OAuth)
- `src/middleware.ts` (the admin gate + UTM capture)
- dynamic routes (`src/pages/[area].astro`, `for-[persona].astro`, `blog/[...slug].astro`)
- anything that logs, sends to a third party, or renders untrusted strings

Check each box only after you've **read the actual code path end-to-end**, not
just the diff. An item you can't verify is a finding, not a pass.

## Scope note: mobile terms → this stack

This review was requested in terms of _platform-channel validation_,
_deep-link routing_, and _sensitive-data logging in crash reports_. Those are
mobile/Flutter concepts; this repo has **no** native/mobile code (it's Astro 5 +
React 19 + Vercel serverless). Each category below is the true web-stack
equivalent of the requested area.

| Requested (mobile)                      | Equivalent here                                                      | Where to look                                                                                                                                                    |
| --------------------------------------- | -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Platform channel validation             | Untrusted→trusted boundary: API/webhook input validation, HMAC, auth | `src/pages/api/lead.ts`, `src/pages/api/callrail-webhook.ts`, `src/lib/callrail.ts`, `src/pages/api/auth/callback.ts`                                            |
| Deep link routing                       | Dynamic routing, redirects, OAuth return-path / open-redirect        | `src/pages/[area].astro`, `for-[persona].astro`, `blog/[...slug].astro`, `src/pages/api/auth/login.ts` + `callback.ts`, `src/pages/api/keystatic/[...params].ts` |
| Sensitive data logging in crash reports | PII in server logs, error pages, analytics dataLayer                 | `src/pages/api/callrail-webhook.ts`, `src/pages/api/lead.ts`, `src/lib/notify.ts`, `src/lib/analytics.ts`                                                        |

## 1. Boundary & input validation (≙ "platform channel validation")

The serverless API routes are the trust boundary between anything on the public
internet and our CRM/SMS/email side effects — the web equivalent of a platform
channel. AI loves to wire up the happy path and skip the guard.

- [ ] **Every mutating endpoint checks intent/auth before side effects.** No
      `POST`/`PUT`/`DELETE` handler runs a write or an outbound call before
      validating the request. _Why AI misses it:_ in a scan of AI repos, **38%**
      had missing auth on sensitive routes and **52% of Bolt.new repos** had at
      least one unprotected mutating endpoint — the model generates the handler
      body and forgets the gate.
- [ ] **Webhooks verify the signature _before_ parsing or acting.** In
      `callrail-webhook.ts` the HMAC check (`verifyCallRailSignature`) runs and
      returns `401` _before_ `JSON.parse`, and returns `503` (fail-closed) when
      `CALLRAIL_WEBHOOK_SECRET` is unset. Confirm the comparison stays
      constant-time (`timingSafeEqual` in `src/lib/callrail.ts`) — AI often
      "simplifies" this to `===`, which is timing-attackable.
- [ ] **Server-side validation exists independent of the browser.** The
      `lead.ts` required-field check (`firstName`, `phone`, `address`, `city`,
      `zip`, `consent` → `422`) must not be the HTML5 `pattern`/`required`
      attributes alone. _Why AI misses it:_ it treats client-side validation as
      sufficient. Re-validate type/shape of any JSON body cast to `FormShape`.
- [ ] **No SSRF via request-derived URLs.** Any `fetch()` whose host/path comes
      from request data must be pinned to a known host. None should be reachable
      today — confirm new code doesn't introduce one. Beware the subtler
      variants: a `3xx` redirect to an internal address and DNS-rebinding to
      `169.254.169.254` both bypass naive URL allowlists.
- [ ] **No secret leaks through a `PUBLIC_*` env var or the client bundle.**
      Only non-sensitive values may use the `PUBLIC_` prefix (the Astro analogue
      of Next's `NEXT_PUBLIC_` footgun). _Why AI misses it:_ it slaps the public
      prefix on a var to "make it work" client-side, shipping the secret to every
      browser. Scanned AI repos had hardcoded secrets **45%** of the time.
- [ ] **Honeypot + rate-limit are present _and_ understood as best-effort.** The
      `website` honeypot (silent `303`) and `rateLimit()` are real defenses, but
      the in-memory limiter in `src/lib/rate-limit.ts` **resets on every cold
      start** and is per-instance — it is not a distributed control. Don't let a
      reviewer (or the AI's comment) claim it's a hard guarantee. A real cap
      needs a shared store (Vercel KV / Upstash).
- [ ] **Client IP key isn't blindly trusted.** `clientKey()` takes the first
      `x-forwarded-for` segment, which a client can spoof to dodge the rate
      limit. Acceptable for best-effort today; flag if any _security_ decision
      starts depending on it.

## 2. Routing & redirects (≙ "deep link routing")

URLs that drive navigation, content selection, or post-auth redirects are the
web's deep links. The classic AI mistakes here are runtime lookups from raw
input and string-prefix redirect checks.

- [ ] **Dynamic routes resolve against an allowlist, never raw input.**
      `[area]`, `for-[persona]`, and `blog/[...slug]` all enumerate valid paths
      via `getStaticPaths()` against hardcoded data
      (`SERVICE_AREA_SLUGS`, persona slugs, published collection entries). No
      route param is used to index into data or build a path at request time.
- [ ] **Redirect targets are validated as same-site.** `safeReturnTo()` in
      `login.ts` (and the inline check at `callback.ts:131`) require a leading
      `/` and reject `//` — confirm any new redirect uses the same rule (ideally
      `new URL()`), **not** a `startsWith(host)` string check. _Why it matters:_
      open redirect → the OAuth `code`/token gets forwarded to an
      attacker-controlled host (token theft). Validate with parsing, not string
      ops.
- [ ] **OAuth CSRF `state` is set _and_ verified.** `login.ts` sets the
      `nv_oauth_state` cookie; `callback.ts` rejects when it's missing or
      doesn't match (`bad-state`). Confirm both halves survive any refactor — AI
      sometimes keeps the cookie set and drops the verify.
- [ ] **`redirect_uri` is pinned to a trusted origin, not the request host.**
      Both routes build it from `publicOrigin(request)`
      (`src/lib/site-url.ts`), not from a user-controllable header. A
      host-header-derived `redirect_uri` is an open-redirect/code-interception
      vector.
- [ ] **Catch-all routes can't be coerced to unintended handlers.** The
      `keystatic/[...params]` wrapper and `blog/[...slug]` must not let a crafted
      path reach an unexpected branch or skip the admin gate in
      `src/middleware.ts`. Trace a hostile path through middleware → route.

## 3. Logging, errors & PII (≙ "sensitive data in crash reports")

Server logs and error responses are where this app's "crash reports" live. The
recurring AI failure is logging the whole request/object — PII and all — for
debugging and never removing it.

- [ ] **No PII in `console.*`, thrown errors, or analytics.** **Known finding:**
      `src/pages/api/callrail-webhook.ts` logs `customer_phone` (and
      `tracking_phone`) on every call — a customer phone number written to Vercel
      function logs. Treat this as the worked example: prefer redaction/hashing
      at the call site, or log only an event id. (Fixing it is tracked
      separately; this checklist flags it so new code doesn't repeat the
      pattern.)
- [ ] **`lead.ts` / `notify.ts` keep PII out of logs.** On failure they log
      `hl.error` and `notify.errors` — confirm those strings never interpolate
      the lead's name/phone/email/address. Redact PII as early as possible
      (at the shipper/source), per logging best practice.
- [ ] **Error responses leak nothing.** `callback.ts` wraps the handler so an
      exception renders a readable page instead of `FUNCTION_INVOCATION_FAILED`,
      and the message is escaped (`escapeHtml`). Confirm no stack trace, env var,
      token, or secret reaches the response body or a `5xx` page.
- [ ] **Analytics payloads carry only non-PII.** `src/lib/analytics.ts` pushes
      conversion fields (`event`, `value`, `currency`, `transaction_id`) to the
      dataLayer — never name/phone/email. Any new `dataLayer.push` gets the same
      scrutiny.
- [ ] **Third-party failures don't log credentials.** When HighLevel/Twilio/
      Resend calls fail (`highlevel.ts`, `notify.ts`), the API key/auth token
      must never appear in the logged error.

## 4. Cross-cutting AI-codegen smells

- [ ] **`set:html` / `dangerouslySetInnerHTML` only on author-controlled
      constants.** Today `set:html` is used in `ServiceLayout.astro` on
      hardcoded strings from `src/lib/services.ts` — safe. Never point it at
      request data, CMS/Keystatic content, or anything user-derived without
      sanitization. _Why AI misses it:_ `dangerouslySetInnerHTML` appears in
      ~28% of React codebases handling user content and **fewer than 10%**
      sanitize; XSS is ~21% of all web-app vulns (Veracode 2024). Also reject
      unvalidated `href` that could be a `javascript:` URI.
- [ ] **Verify the whole path, not the diff.** Because the model writes one file
      at a time, re-read auth flow → route → middleware together. A change that
      looks correct in isolation can silently bypass the admin gate or drop a
      check upstream.
- [ ] **Dependency/version sanity.** Confirm new/changed deps aren't on a
      known-bad release (React/Next/Astro framework CVEs have landed fast and
      often recently). Check pinned versions against current advisories.
- [ ] **No invented behavior or fabricated config.** Per repo Hard Rules, AI
      must not invent env vars, endpoints, or "helpful" data. Anything it can't
      source should be a visible `NEEDS DATA`/`TODO`, not a confident guess.
